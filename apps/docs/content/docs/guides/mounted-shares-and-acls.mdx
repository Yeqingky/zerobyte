title: 挂载共享与权限
description: 了解挂载卷的优势、简化之处以及何时更适合使用本地目录
---

挂载卷在 Zerobyte 中仍然非常有用。对许多人来说，备份文件内容是主要目标，挂载共享在这方面表现良好。

权限和其他文件系统细节则稍微复杂一些。

<Callout type="info">
  备份挂载共享并**不会**从源端移除权限。

  主要问题比听起来更简单：挂载的视图能向备份展示多少源系统的权限信息？
</Callout>

## 简要总结

- 如果您的主要目标是恢复文件和文件夹，挂载卷通常完全可以满足需求。
- Zerobyte 备份的是容器内挂载文件系统所呈现的视图。
- 某些后端能够很好地显示权限和所有权。
- 某些后端则显示简化后的视图。
- 如果您想要最接近真实副本的效果，本地**目录**卷是最佳选择。

## 为什么这很重要

当 Zerobyte 备份远程卷时，它并不直接以原生格式与每个存储系统通信。

实际情况是：

1. Zerobyte 将卷挂载到 Linux 容器内。
2. Linux 将挂载的卷视为普通文件系统。
3. Restic 备份 Linux 在该文件系统上能看到的内容。

这意味着备份的详细信息程度取决于挂载视图的详细程度。

如果挂载视图显示真实的拥有者、权限和高级权限规则，Restic 就可以备份它们。

如果挂载视图显示的是简化版本，快照也将包含该简化版本。

## 可能被简化的内容

根据后端不同，挂载视图可能简化以下内容：

- 文件拥有者
- 组
- 读、写和执行权限
- 高级权限规则
- 额外的文件系统元数据

这就是为什么备份在文件内容方面可能非常出色，但在权限细节方面可能不够精确。

## 在挂载共享上编辑文件

这在备份之外也很有用。

- 编辑现有文件通常会保留该文件原有的服务器端权限。
- 删除文件并重新创建通常会产生一个新的远程文件。
- 新文件通常会获得文件夹的默认或继承权限，不一定是旧文件的精确权限。
- 某些编辑器通过替换文件来保存，因此它们的行为更像是重新创建而非原地编辑。

对许多用户来说这不是问题。这主要在你试图保持权限完全一致时才重要。

权限更重要的常见示例：

- 共享的公司文件夹，不同团队应看到不同的子文件夹
- 用户主目录，私人文件在恢复后应保持私密
- 网站或应用文件，其服务依赖于正确的拥有者或可执行权限
- 服务器脚本、cron 任务或部署钩子，需要保持可执行状态
- 具有继承文件夹规则的业务共享，例如财务或人力资源文档

## 后端对比

| 后端 | 适用场景 | 权限保真度 | 注意事项 |
| --- | --- | --- | --- |
| **目录** | 本地数据和绑定挂载的主机路径 | **最佳** | 当你希望备份和恢复尽可能接近原始文件系统时的最佳选择 |
| **NFS** | Linux 和 NAS 共享 | **通常良好** | 通常是较强的远程选项，但所有权映射在不同系统之间仍可能有差异 |
| **SMB/CIFS** | Windows 共享和 Samba 服务器 | **中等至良好** | 当权限细节很重要时通常是最佳的远程选项，但仍不是原始 Windows 权限模型的完美副本 |
| **WebDAV** | Nextcloud、ownCloud 和面向内容的远程访问 | **有限** | 非常适合备份文件，但在精确权限复制方面可靠性较低 |
| **SFTP** | 安全访问远程 Linux 服务器 | **有限** | 适合内容备份，但高级权限细节通常会被简化 |
| **Rclone** | 云提供商和对象存储 | **最低** | 最好视为内容源，而非完整的文件系统副本 |

## 各后端的优势

### 目录

当您同时关心文件内容和权限时，这是最容易信赖的选项。

为什么它表现如此出色：

- Zerobyte 直接访问文件系统
- 没有额外的网络协议来转换数据
- 恢复通常是可预测性最高的

如果您想要最接近真实副本的效果，尽可能选择**目录**。

### NFS

NFS 通常非常适合 Linux 风格的文件共享。

在以下情况下它通常是强有力的选择：

- 共享来自 Linux 或 NAS 系统
- 系统之间的所有权匹配良好
- 您希望一个仍然相当原生的远程选项

根据服务器配置方式，结果可能有所不同，因此最好先用几个真实文件进行测试。

### SMB/CIFS

SMB 是 Windows 共享和 Samba 服务器的不错选择。

与其他远程选项相比，它通常能为 Linux 提供更丰富的远程权限视图。当权限细节很重要时，它是最有希望的远程选择。

不过，最好将 SMB 视为"通常良好"而非"保证精确"。

### WebDAV

WebDAV 非常方便且被广泛支持。

它通常最适合：

- 备份文件内容
- 访问 Nextcloud 和 ownCloud 等服务
- 简单的远程访问

如果您想要非常精确的原始权限模型副本，它通常**不是**最佳选择。

### SFTP

SFTP 是当您需要通过 SSH 进行安全远程访问时的可靠选择。

它通常适用于：

- 从远程 Linux 服务器备份文件
- 使用密码或密钥进行简单远程访问
- 面向内容的备份

与 WebDAV 一样，它通常更适合文件内容而非精确权限复制。

### Rclone

Rclone 在连接众多云提供商方面表现出色。

它通常最适合：

- 备份存储在云服务中的数据
- 面向内容的备份作业
- 非传统文件系统的提供商

当您希望挂载视图表现得像普通本地文件系统时，它是最不合适的选择。

## 真正复制的最佳选择

如果您所说的"真正复制"是指：

- 相同的文件
- 相同的所有权
- 相同的权限
- 相同的高级权限规则
- 恢复行为与原始保持高度一致

那么最佳选择仍然是**目录**。

如果原始数据位于其他地方，最可靠的方法通常是以下几种之一：

1. 在数据附近运行 Zerobyte，并将其作为本地目录备份。
2. 先在主机上挂载源，然后将该本地路径作为目录卷绑定挂载到 Zerobyte 中。
3. 将远程挂载后端主要用于内容备份，而非精确复制。

## 实用建议

1. 如果您主要关心恢复文件，大多数挂载后端都可以。
2. 如果您也非常关心权限，首选**目录**。
3. 在**目录**之后，**NFS** 通常是次优选择。
4. **SMB** 也是一个不错的选择，尤其是对于 Windows 和 Samba 共享，但值得先进行测试。
5. 将 **WebDAV**、**SFTP** 和 **rclone** 视为内容优先的后端。

<Callout type="info">
  这里的信息不是"避免使用挂载共享"。它们很有用，而且常常是恰当的选择。

  更简单的规则是：如果文件内容最重要，挂载共享通常很出色。如果您想要尽可能接近原始文件系统的副本，本地**目录**卷仍然是最佳选择。
</Callout>
