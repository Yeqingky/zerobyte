---
title: 备份 Webhook
description: 在备份任务前后运行 HTTP 钩子
---

备份 Webhook 让备份作业在 Restic 启动前和完成后立即调用 HTTP 端点。当源端需要在备份前后执行短时运行时操作时使用，例如暂停服务、创建数据库转储、刷新缓存或在快照完成后恢复容器。

备份 Webhook 在每个备份作业的**高级**部分中配置。它们与[通知](/docs/guides/notifications)不同：通知将备份事件报告给人员或系统，而备份 Webhook 是备份执行生命周期的一部分。

## 备份 Webhook 的工作原理

Zerobyte 支持两个生命周期钩子：

| 钩子 | 运行时机 | 失败行为 |
| --- | --- | --- |
| **备份前 Webhook** | 在 Restic 开始读取卷之前 | 请求失败会在 Restic 运行前停止备份 |
| **备份后 Webhook** | 在 Restic 完成、失败或被取消后 | 失败的请求会记录在最终结果中；原本正常的备份会变为警告 |

每个钩子发送一个 `POST` 请求。返回 `2xx` 状态码视为成功。不跟随重定向。Webhook 请求在 `WEBHOOK_TIMEOUT` 秒后超时，默认为 60 秒。

<Callout type="warn">
  每个备份 Webhook URL 必须使用 `WEBHOOK_ALLOWED_ORIGINS` 中列出的来源。来源是协议、主机名和端口，例如 `http://host.docker.internal:9000`。
</Callout>

## 请求体

如果钩子体字段为空，Zerobyte 会发送一个 JSON 备份上下文体，并设置 `Content-Type: application/json`。

备份前 Webhook 示例：

```json
{
  "phase": "pre",
  "event": "backup.pre",
  "jobId": "job_...",
  "scheduleId": "sched_...",
  "organizationId": "org_...",
  "sourcePath": "/data"
}
```

备份后 Webhook 示例：

```json
{
  "phase": "post",
  "event": "backup.post",
  "jobId": "job_...",
  "scheduleId": "sched_...",
  "organizationId": "org_...",
  "sourcePath": "/data",
  "status": "success"
}
```

`status` 仅发送给备份后 Webhook。它可以是 `success`、`warning`、`error` 或 `cancelled`。当 Zerobyte 需要报告警告、失败或取消详情时，`error` 会包含在备份后 Webhook 中。

如果您输入自定义体，Zerobyte 将发送该确切体而不是默认的 JSON 上下文。如果接收方需要，请自行添加 `Content-Type` 标头。

## 标头

标头是可选的，每行输入一个：

```text
X-Zerobyte-Hook-Secret: replace-with-a-long-random-secret
Content-Type: application/json
```

标头值以明文形式存储。请使用作用域限定的 Webhook 密钥，而不是可重复使用的账户密码或长期基础设施令牌。

## 配置备份钩子

1. 将 Webhook 来源添加到 Zerobyte 环境变量 `WEBHOOK_ALLOWED_ORIGINS` 中。
2. 重启 Zerobyte 以加载环境变量更改。
3. 打开**备份**并选择备份作业。
4. 编辑作业并展开**高级**。
5. 填写**备份前 Webhook** 或**备份后 Webhook**。
6. 添加任何必需的标头。
7. 除非接收服务需要自定义负载，否则将体字段留空。
8. 保存备份作业并运行**立即备份**以测试生命周期。

对于 Linux 上的 Docker Compose，`host.docker.internal` 通常需要显式的主机网关条目：

```yaml docker-compose.yml
services:
  zerobyte:
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      - WEBHOOK_ALLOWED_ORIGINS=http://host.docker.internal:9000
```

## 实践：使用 adnanh/webhook 启停 Postgres 容器

此示例在 Docker 主机上运行 [`adnanh/webhook`](https://github.com/adnanh/webhook)。Zerobyte 在备份前后调用它：

- 备份前钩子停止 `postgres` 容器。
- Restic 备份已挂载的数据。
- 备份后钩子重新启动 `postgres` 容器。

<Callout type="warn">
  停止数据库容器是一种粗暴的一致性策略。仅在可接受短暂停机时使用。对于大型数据库，建议使用原生数据库转储、复制快照或存储级快照。
</Callout>

### 1. 在 Docker 主机上安装 webhook

在 Debian 或 Ubuntu 上：

```bash
sudo apt-get update
sudo apt-get install webhook
```

`webhook` 在 `/hooks/<hook-id>` 提供已配置的钩子。默认端口为 `9000`，`-hooks` 标志指向 JSON 或 YAML 钩子文件。

### 2. 创建钩子脚本

为脚本创建目录：

```bash
sudo mkdir -p /opt/zerobyte-hooks
```

创建 `/opt/zerobyte-hooks/stop-postgres.sh`：

```sh
#!/bin/sh
set -eu

CONTAINER=postgres

STATE=$(docker inspect -f '{{.State.Running}}' "$CONTAINER")

if [ "$STATE" = "true" ]; then
  docker stop "$CONTAINER"
fi
```

创建 `/opt/zerobyte-hooks/start-postgres.sh`：

```sh
#!/bin/sh
set -eu

CONTAINER=postgres

STATE=$(docker inspect -f '{{.State.Running}}' "$CONTAINER")

if [ "$STATE" != "true" ]; then
  docker start "$CONTAINER"
fi
```

使两个脚本可执行：

```bash
sudo chmod +x /opt/zerobyte-hooks/stop-postgres.sh /opt/zerobyte-hooks/start-postgres.sh
```

如果您的容器名称不同，请更改两个脚本中的 `CONTAINER=postgres`。

### 3. 创建 Webhook 配置

创建 `/opt/zerobyte-hooks/hooks.json`：

```json
[
  {
    "id": "stop-postgres",
    "execute-command": "/opt/zerobyte-hooks/stop-postgres.sh",
    "command-working-directory": "/opt/zerobyte-hooks",
    "http-methods": ["POST"],
    "include-command-output-in-response": true,
    "trigger-rule": {
      "match": {
        "type": "value",
        "value": "replace-with-a-long-random-secret",
        "parameter": {
          "source": "header",
          "name": "X-Zerobyte-Hook-Secret"
        }
      }
    }
  },
  {
    "id": "start-postgres",
    "execute-command": "/opt/zerobyte-hooks/start-postgres.sh",
    "command-working-directory": "/opt/zerobyte-hooks",
    "http-methods": ["POST"],
    "include-command-output-in-response": true,
    "trigger-rule": {
      "match": {
        "type": "value",
        "value": "replace-with-a-long-random-secret",
        "parameter": {
          "source": "header",
          "name": "X-Zerobyte-Hook-Secret"
        }
      }
    }
  }
]
```

在两个钩子定义中使用相同的密钥。`include-command-output-in-response` 使 `webhook` 等待脚本执行完成，并在命令失败时返回错误响应，这允许 Zerobyte 在备份前钩子无法停止 Postgres 时停止备份。

### 4. 启动 Webhook

首先在前台运行：

```bash
sudo webhook -hooks /opt/zerobyte-hooks/hooks.json -port 9000 -verbose -http-methods POST
```

在另一个 shell 中测试两个钩子：

```bash
curl -X POST \
  -H "X-Zerobyte-Hook-Secret: replace-with-a-long-random-secret" \
  http://localhost:9000/hooks/stop-postgres

curl -X POST \
  -H "X-Zerobyte-Hook-Secret: replace-with-a-long-random-secret" \
  http://localhost:9000/hooks/start-postgres
```

测试通过后，使用常规进程管理器运行 `webhook`。

### 5. 允许 Zerobyte 调用 Webhook 服务器

将 Webhook 服务器来源添加到 Zerobyte：

```yaml docker-compose.yml
services:
  zerobyte:
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      - WEBHOOK_ALLOWED_ORIGINS=http://host.docker.internal:9000
```

重启 Zerobyte：

```bash
docker compose up -d
```

### 6. 将钩子添加到备份作业

在 Zerobyte 中打开备份作业，编辑它，并展开**高级**。

使用以下值：

```text
备份前 Webhook: http://host.docker.internal:9000/hooks/stop-postgres
备份前 Webhook 标头:
X-Zerobyte-Hook-Secret: replace-with-a-long-random-secret

备份后 Webhook: http://host.docker.internal:9000/hooks/start-postgres
备份后 Webhook 标头:
X-Zerobyte-Hook-Secret: replace-with-a-long-random-secret
```

将两个体字段留空。Zerobyte 将发送默认的 JSON 上下文体。

运行**立即备份**。如果停止钩子失败或返回非 `2xx` 响应，Zerobyte 会在 Restic 启动前使备份失败。如果启动钩子在 Restic 完成后失败，Zerobyte 会将问题记录在运行详情中，以便您可以手动重启容器。

### 7. 将 Webhook 作为服务运行

前台测试通过后，创建一个小的 systemd 单元，使 `webhook` 在启动时自动运行。

创建 `/etc/systemd/system/zerobyte-webhook.service`：

```ini
[Unit]
Description=Zerobyte backup webhook runner
After=network-online.target docker.service
Wants=network-online.target
Requires=docker.service

[Service]
Type=simple
ExecStart=/usr/bin/webhook -hooks /opt/zerobyte-hooks/hooks.json -port 9000 -http-methods POST -verbose
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
```

启用并启动它：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now zerobyte-webhook.service
sudo systemctl status zerobyte-webhook.service
```

查看日志：

```bash
sudo journalctl -u zerobyte-webhook.service -f
```
