Zerobyte 可以在启动时从 JSON 配置文件同步由运维管理的仓库和卷。当您希望凭据和连接信息存在于部署时配置中而不是通过 UI 输入时，这非常有用。

## 预配当前支持的内容

预配当前支持：

- 仓库
- 卷
- 通过 `env://` 和 `file://` 的密钥引用
- 通过保持相同的 `id` 在原地更新已管理的资源
- 使用 `"delete": true` 删除已管理的资源

预配当前**不**支持：

- 备份任务或计划
- 通知目的地
- 将 UI 创建的资源导出为预配 JSON
- 无需重启 Zerobyte 容器的热重载

<Callout type="warn">
	预配仅限于 schema 版本 `1` 的仓库和卷。如果您需要管理备份任务或通知，请暂时通过 UI 或 API 创建。
</Callout>

## 前提条件

- 一个正在运行的 Zerobyte 实例，且拥有组织 ID
- 指向您的 JSON 文件的 `PROVISIONING_PATH` 环境变量

### 查找您的组织 ID

1. 登录 Zerobyte 并切换到您要预配到的组织
2. 打开 **设置**
3. 打开 **组织** 标签页
4. 从 **组织详情** 中复制只读的 **组织 ID** 值

<Callout type="info">
	**组织** 标签页仅对组织中的 `admin`（管理员）和 `owner`（所有者）可见。如果您看不到它，请向组织管理员或所有者询问 ID，或请他们先授予您所需的角色。
</Callout>

## 快速开始

<Steps>
<Step>

### 创建预配文件

创建一个 `provisioning.json` 文件：

```json provisioning.json
{
	"version": 1,
	"repositories": [
		{
			"id": "local-repo",
			"organizationId": "your-organization-id",
			"name": "Primary Local Repository",
			"backend": "local",
			"compressionMode": "auto",
			"config": {
				"backend": "local",
				"path": "/var/lib/zerobyte/repositories/primary",
				"isExistingRepository": false
			}
		}
	],
	"volumes": [
		{
			"id": "documents",
			"organizationId": "your-organization-id",
			"name": "Documents",
			"backend": "directory",
			"autoRemount": true,
			"config": {
				"backend": "directory",
				"path": "/data/documents"
			}
		}
	]
}
```

</Step>
<Step>

### 挂载文件并配置 Zerobyte

挂载预配文件并设置 `PROVISIONING_PATH`：

```yaml docker-compose.yml
services:
  zerobyte:
    image: ghcr.io/nicotsx/zerobyte:latest
    environment:
      - PROVISIONING_PATH=/config/provisioning.json
    volumes:
      - ./provisioning.json:/config/provisioning.json:ro
      - /var/lib/zerobyte:/var/lib/zerobyte
      - /srv/documents:/data/documents
```

</Step>
<Step>

### 启动或重启 Zerobyte

```bash
docker compose up -d
```

启动时，Zerobyte 会读取预配文件、解析支持的密钥引用、加密解析后的密钥值，并将已管理的资源同步到数据库中。

</Step>
</Steps>

## 根文件格式

根对象始终使用相同的结构：

```json
{
	"version": 1,
	"repositories": [],
	"volumes": []
}
```

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `version` | 是 | 必须是 `1` |
| `repositories` | 否 | 已预配的仓库数组。默认为 `[]` |
| `volumes` | 否 | 已预配的卷数组。默认为 `[]` |

## 仓库条目参考

`repositories` 中的每个条目使用以下顶层结构：

```json
{
	"id": "repo-id",
	"organizationId": "org-id",
	"name": "Repository Name",
	"backend": "local",
	"compressionMode": "auto",
	"delete": false,
	"config": {
		"backend": "local",
		"path": "/var/lib/zerobyte/repositories/repo-id"
	}
}
```

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 组织内部的稳定预配标识符。如果您希望 Zerobyte 在原地更新同一资源，请保持此值稳定 |
| `organizationId` | 是 | 必须与现有组织匹配 |
| `name` | 是 | 在 Zerobyte 中显示的名称 |
| `backend` | 是 | 必须是 `local`、`s3`、`r2`、`gcs`、`azure`、`rclone`、`rest`、`sftp` 之一 |
| `compressionMode` | 否 | `off`、`auto` 或 `max` |
| `delete` | 否 | 默认为 `false` |
| `config` | 是 | 特定于后端的对象。`config.backend` 必须与顶层的 `backend` 值匹配 |

### 仓库共享配置字段

以下字段在每个仓库的 `config` 对象中均可使用：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `isExistingRepository` | 否 | 为 `true` 时，Zerobyte 将该仓库视为已初始化。省略或为 `false` 时，新创建的预配仓库会在首次同步时使用 `restic init` 进行初始化 |
| `customPassword` | 否 | 覆盖组织级别的仓库密码 |
| `cacert` | 否 | 自定义 CA 证书内容 |
| `insecureTls` | 否 | 禁用支持的后端的 TLS 验证 |
| `uploadLimit` | 否 | 带宽限制对象 |
| `downloadLimit` | 否 | 带宽限制对象 |

带宽限制对象使用以下结构：

```json
{
	"enabled": true,
	"value": 10,
	"unit": "Mbps"
}
```

`unit` 必须是 `Kbps`、`Mbps` 或 `Gbps` 之一。

## 卷条目参考

`volumes` 中的每个条目使用以下顶层结构：

```json
{
	"id": "volume-id",
	"organizationId": "org-id",
	"name": "Volume Name",
	"backend": "directory",
	"autoRemount": true,
	"delete": false,
	"config": {
		"backend": "directory",
		"path": "/data/volume"
	}
}
```

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 组织内部的稳定预配标识符 |
| `organizationId` | 是 | 必须与现有组织匹配 |
| `name` | 是 | 在 Zerobyte 中显示的名称 |
| `backend` | 是 | 必须是 `nfs`、`smb`、`directory`、`webdav`、`rclone`、`sftp` 之一 |
| `autoRemount` | 否 | 默认为 `true` |
| `delete` | 否 | 默认为 `false` |
| `config` | 是 | 特定于后端的对象。`config.backend` 必须与顶层的 `backend` 值匹配 |

## 密钥引用

支持的密钥引用使用以下前缀之一：

| 引用 | 解析来源 | 示例 |
| --- | --- | --- |
| `env://VARIABLE_NAME` | 容器环境变量 | `env://AWS_ACCESS_KEY_ID` |
| `file://secret_name` | `/run/secrets/secret_name` | `file://aws_secret_access_key` |

<Callout type="info">
	`file://` 引用始终从 `/run/secrets/` 解析，且必须是一个单独的文件名，不能是嵌套路径。
</Callout>

只有特定字段会被解析为密钥引用。其他字符串字段将按字面值处理。

### 支持密钥引用的仓库字段

- 共享字段：`customPassword`、`cacert`
- `s3`：`accessKeyId`、`secretAccessKey`
- `r2`：`accessKeyId`、`secretAccessKey`
- `gcs`：`credentialsJson`
- `azure`：`accountKey`
- `rest`：`username`、`password`
- `sftp`：`privateKey`

### 支持密钥引用的卷字段

- `smb`：`password`
- `webdav`：`password`
- `sftp`：`password`、`privateKey`

解析后的密钥值会在 Zerobyte 存储到数据库之前进行加密。

## 轮换密钥

要轮换预配的密钥：

1. 更新环境变量或密钥文件
2. 使用 `docker compose restart` 重启 Zerobyte

Zerobyte 在每次启动时都会重新解析支持的密钥引用。

## 更新和删除已管理的资源

- 保持相同的 `id` 以在原地更新已管理的资源
- 仅更改 `name` 会保留相同的底层资源记录
- 设置 `"delete": true` 会在下次启动同步时删除已管理的资源

删除条目目前仍需包含完整的条目结构，因为在 Zerobyte 应用 `delete` 标志之前，预配文件会先经过验证：

```json
{
	"id": "repo-to-remove",
	"organizationId": "your-organization-id",
	"name": "Repo to remove",
	"backend": "local",
	"delete": true,
	"config": {
		"backend": "local",
		"path": "/var/lib/zerobyte/repositories/old",
		"isExistingRepository": true
	}
}
```

## 故障排除

### `No matching discriminator`

此错误通常表示以下情况之一存在问题：

- `backend` 不是该资源类型的有效值
- `config.backend` 与顶层的 `backend` 不匹配
- `config` 对象不符合该后端所需的字段要求

### 密钥引用错误

- `env://NAME` 要求该环境变量在 Zerobyte 容器中存在
- `file://name` 要求 `/run/secrets/name` 存在
- Zerobyte 在遇到预配错误时会停止同步，因此请修复第一个报告的错误后重新启动

## 重要说明

<Callout type="warn">
	每个预配条目必须引用一个现有的 `organizationId`。在启用预配之前先完成首次设置，以便您拥有有效的组织 ID。
</Callout>

- 对 `provisioning.json` 的更改仅在容器重启后生效
- 预配的资源会出现在正常 UI 中，并标记为"已管理"
- 在 UI 中编辑预配的资源适用于测试，但下次预配同步可能会覆盖它
