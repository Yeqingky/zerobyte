---
title: 反向代理
description: 在 Nginx、Caddy 或 Traefik 后面通过 HTTPS 运行 Zerobyte
---

在反向代理后面运行 Zerobyte 时，您需要配置 `BASE_URL` 环境变量指向您的 HTTPS 域名，并确保代理传递正确的请求头。

## 前提条件

在您的 `docker-compose.yml` 中设置以下环境变量：

```yaml docker-compose.yml
environment:
  - BASE_URL=https://zerobyte.example.com
  - TRUST_PROXY=true  # 可选：信任来自代理的 X-Forwarded-For 请求头
```

修改后重启容器：

```bash
docker compose down && docker compose up -d
```

## BASE_URL 对 Cookie 安全性的影响

Zerobyte 使用 `BASE_URL` 来确定认证 Cookie 是否应标记为 `Secure`。这直接影响登录会话的工作方式：

| BASE_URL 值 | Cookie 行为 |
|---|---|
| `http://` 或 IP 地址（例如 `http://192.168.1.100:4096`） | Secure Cookie **禁用**，Cookie 通过 HTTP 发送。适用于本地网络访问。 |
| `https://` 域名（例如 `https://zerobyte.example.com`） | Secure Cookie **启用**，Cookie 仅通过 HTTPS 发送。生产环境必须使用此设置。 |

<Callout type="warn">
如果 `BASE_URL` 设置为 `https://` 地址，浏览器将**仅**通过 HTTPS 连接发送认证 Cookie。通过普通 HTTP 访问 Zerobyte 将导致登录循环或会话错误。请确保您的反向代理在将请求转发到 Zerobyte 之前已终止 TLS。
</Callout>

## 代理配置

<Tabs items={["Nginx", "Caddy", "Traefik"]}>
<Tab value="Nginx">

### Nginx

为 Zerobyte 创建一个 server 块。此配置负责 TLS 终止并将请求转发到容器：

```nginx nginx.conf
server {
    listen 443 ssl http2;
    server_name zerobyte.example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://localhost:4096;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

保存后重新加载 Nginx：

```bash
sudo nginx -t && sudo systemctl reload nginx
```

</Tab>

<Tab value="Caddy">

### Caddy

Caddy 会自动从 Let's Encrypt 申请和续期 TLS 证书。将此内容添加到您的 `Caddyfile` 中：

```text Caddyfile
zerobyte.example.com {
    reverse_proxy localhost:4096
}
```

保存后重新加载 Caddy：

```bash
sudo systemctl reload caddy
```

Caddy 会自动处理 `X-Forwarded-For`、`X-Forwarded-Proto` 和 TLS，无需额外配置。

</Tab>

<Tab value="Traefik">

### Traefik

如果您使用 Traefik 作为反向代理，请在 `docker-compose.yml` 中为 Zerobyte 服务添加标签：

```yaml docker-compose.yml
services:
  zerobyte:
    image: ghcr.io/nicotsx/zerobyte:v0.36
    container_name: zerobyte
    restart: unless-stopped
    environment:
      - BASE_URL=https://zerobyte.example.com
      - APP_SECRET=your-secret-here
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.zerobyte.rule=Host(`zerobyte.example.com`)"
      - "traefik.http.routers.zerobyte.entrypoints=websecure"
      - "traefik.http.routers.zerobyte.tls.certresolver=letsencrypt"
      - "traefik.http.services.zerobyte.loadbalancer.server.port=4096"
    volumes:
      - /etc/localtime:/etc/localtime:ro
      - /var/lib/zerobyte:/var/lib/zerobyte
```

确保您的 Traefik 实例配置了 `websecure` 入口点和 `letsencrypt` 证书解析器。

</Tab>
</Tabs>

## 绑定到 localhost

如果您在同一台机器上使用反向代理，请将 Zerobyte 端口绑定到 `127.0.0.1`，使其无法从网络直接访问：

```yaml docker-compose.yml
ports:
  - "127.0.0.1:4096:4096"
```

这样可以确保所有流量都经过您的反向代理，由反向代理处理 TLS 和认证请求头。

## 安全注意事项

<Callout type="warn">
**将 Zerobyte 暴露到互联网需要谨慎。** 即使使用了带有 HTTPS 的反向代理，请确保您：

- 仅将容器端口绑定到 localhost（`127.0.0.1:4096:4096`），使其无法被直接访问。
- 使用强 `APP_SECRET`（使用 `openssl rand -hex 32` 生成）。
- 保持 Zerobyte 更新到最新版本。
- 考虑将 Zerobyte 放在安全隧道（Cloudflare Tunnel、Tailscale、WireGuard）后面，以获得额外的保护层。
</Callout>

### 关于 TRUSTED_ORIGINS 的说明

`TRUSTED_ORIGINS` 环境变量允许您为跨域请求添加额外的 CORS 来源。它**不能**使 HTTP 在 `BASE_URL` 设置为 HTTPS 的情况下正常工作。如果您的 `BASE_URL` 使用 `https://`，无论 `TRUSTED_ORIGINS` 中列出了什么，所有访问都必须通过 HTTPS。

```yaml
# TRUSTED_ORIGINS 仅添加允许的 CORS 来源，不会将 HTTPS 降级为 HTTP
environment:
  - BASE_URL=https://zerobyte.example.com
  - TRUSTED_ORIGINS=https://other-app.example.com
```

## 验证您的设置

配置反向代理后：

<Steps>
<Step>

### 测试 HTTPS 访问

在浏览器中打开 `https://zerobyte.example.com`。您应该能看到 Zerobyte 的登录页面，并附带有效的 TLS 证书。

</Step>

<Step>

### 检查请求头

验证您的代理是否正确转发了请求头。在 Zerobyte 容器日志中，您应该能看到来自代理的请求：

```bash
docker compose logs -f zerobyte
```

</Step>

<Step>

### 确认登录正常

使用您的管理员凭据登录。如果登录失败或出现循环，请仔细检查 `BASE_URL` 是否与您访问的 URL 一致，以及 Cookie 是否正确通过 HTTPS 设置。

</Step>
</Steps>
