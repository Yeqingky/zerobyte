3-2-1 备份策略是降低备份风险的一条简单规则：

- 保留数据的 **3 份副本**
- 使用 **2 种不同的存储系统**
- 保留 **1 份异地副本**

在 Zerobyte 中，通常如下所示：

- 服务器、NAS 或工作站上的实时数据
- 快速本地存储上的主备份仓库
- S3、Cloudflare R2、Google Cloud Storage、Azure Blob Storage、SFTP 或远程 REST 服务器等异地后端的镜像仓库

## 为什么 3-2-1 是个好策略

单一备份副本仍可能因与原始数据相同的原因而失效。3-2-1 规则降低了关联故障的风险，因此一次误删除、磁盘损坏、服务商问题或站点级事故不太可能同时摧毁所有副本。

| 故障场景 | 没有 3-2-1 会出什么问题 | 3-2-1 如何帮助 |
| --- | --- | --- |
| 意外删除或错误变更 | 唯一的备份可能太新或不完整 | 较旧的快照为您提供干净的恢复点 |
| 磁盘或 NAS 故障 | 数据和备份可能同时丢失 | 第二个存储系统仍然保留数据 |
| 勒索软件或主机被入侵 | 本地存储也可能受到影响 | 异地副本为您提供另一条恢复路径 |
| 火灾、盗窃或电源事故 | 同一栋楼里的所有东西可能一起消失 | 异地副本得以幸存 |
| 服务商或账户问题 | 一个后端故障阻止恢复 | 不同的后端减少了单点故障 |

Zerobyte 非常适合此策略，因为它已经为您提供了构建模块：

- **加密仓库**，确保异地副本保持私密
- **增量、去重快照**，使保留多个副本更节省存储空间
- **镜像仓库**，让一个备份任务可以将快照复制到其他目标
- **可浏览的恢复**，让您无需使用命令行即可测试恢复

## 3-2-1 在 Zerobyte 中的映射

| 3-2-1 要素 | Zerobyte 示例 |
| --- | --- |
| **3 份副本** | 实时数据 + 主仓库 + 镜像仓库 |
| **2 种不同的存储系统** | 主副本使用本地磁盘或 NAS，第二副本使用对象存储或远程服务器 |
| **1 份异地副本** | 位于另一栋建筑、区域或服务商的仓库 |

<Callout type="info">
  从历史上看，3-2-1 中的"2"指的是两种不同的介质类型。在现代环境中，不同的存储系统和故障域比字面意义上的介质类型更重要。本地磁盘加云对象存储是一种有力的实际解读。
</Callout>

## 如何在 Zerobyte 中实现

<Steps>
  <Step>

### 为实时数据创建 Volume

将要保护的目录或远程共享添加为 [Volume](/docs/concepts/volumes)。

  </Step>

  <Step>

### 为快速恢复创建主仓库

使用靠近源且速度快的后端，例如本地磁盘、已挂载的 NAS 路径、附近的 REST 服务器或附近的 SFTP 主机。这为您提供了快速的日常恢复目标。

  </Step>

  <Step>

### 在不同故障域中创建第二个仓库

在异地后端上创建另一个仓库，例如 S3 兼容存储、Cloudflare R2、Google Cloud Storage、Azure Blob Storage，或位于其他位置的远程 SFTP 或 REST 服务器。

如果您想要实现 3-2-1 中强有力的"2"，请避免将两个仓库放在同一台主机、同一个机架或同一个服务商账户下。

  </Step>

  <Step>

### 创建备份任务并将异地副本添加为镜像

将主仓库设置为主备份仓库，然后将异地仓库配置为镜像仓库。每次成功备份后，Zerobyte 会自动将该快照复制到镜像仓库。

  </Step>

  <Step>

### 添加调度和保留策略

选择与数据变更频率相匹配的调度计划。对于许多工作负载来说，一个可靠的起点是：

- 调度：`0 2 * * *`（每天凌晨 2:00）
- 保留每日：`7`
- 保留每周：`4`
- 保留每月：`6`
- 保留每年：`1`

根据您的恢复需求和存储预算进行调整。

  </Step>

  <Step>

### 将恢复密钥存储在 Zerobyte 之外

将组织的恢复密钥保存在密码管理器、密钥保管库或 Zerobyte 服务器之外的其他安全位置。异地副本只有在发生故障时仍能解密才有帮助。

  </Step>

  <Step>

### 运行手动备份并测试恢复

使用**立即备份**创建第一个快照，然后测试恢复到非生产路径。只有在验证恢复可行之后，备份策略才真正成立。

  </Step>
</Steps>

## Zerobyte 布局示例

| 副本 | 示例 |
| --- | --- |
| 实时数据 | 服务器上的 `/srv/data` |
| 本地备份副本 | `/mnt/backup-disk/zerobyte` 上的本地仓库 |
| 异地备份副本 | Cloudflare R2 存储桶或远程 SFTP 仓库 |

然后一个备份任务可以写入本地仓库，并将每个成功快照镜像到异地仓库。

## 常见错误

- 将实时数据和主仓库放在同一块物理磁盘上
- 称第二份副本为"异地"，实际上它仍在同一栋楼里
- 将所有副本存储在同一服务商账户下，没有任何隔离
- 忘记保护恢复密钥或导入的仓库密码
- 从未测试过恢复
