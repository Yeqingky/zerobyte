Zerobyte 将**账户认证**与 **Restic 仓库加密**分开处理。您的登录密码、组织的恢复密钥、任何导入的仓库密码以及 `APP_SECRET` 各自服务于不同的目的。

## Zerobyte 中的重要密钥

| 密钥 | 用途 | 适用范围 | 能否从 UI 下载？ |
| --- | --- | --- | --- |
| 用户密码 | 登录 Zerobyte 并确认敏感操作 | 您的用户账户 | 否 |
| 恢复密钥（`restic.pass`） | 打开使用组织默认 Restic 密码的仓库 | 当前活跃的组织 | 是，组织拥有者和管理员在密码重新认证后可以 |
| 自定义仓库密码 | 打开使用不同 Restic 密码的导入仓库 | 单个导入的仓库 | 无单独的下载流程 |
| `APP_SECRET` | 加密 Zerobyte 存储在数据库中的密钥 | 应用部署实例 | 否 |

## 恢复密钥实际上是什么

下载的 `restic.pass` 文件包含**当前活跃组织的 Restic 密码**。

该组织级别的密码是 Zerobyte 在以下情况使用的密码：

- 从 UI 初始化全新的仓库
- 访问依赖组织默认密码的仓库
- 对这些仓库执行常规的备份、恢复、快照、诊断和统计操作

<Callout type="warn">
  将恢复密钥存储在 Zerobyte 之外的密码管理器、密钥库或其他加密存储中。如果您同时丢失了服务器访问权限和正确的 Restic 密码，您的备份将无法恢复。
</Callout>

## 本地用户的首次运行行为

在首次运行引导过程中，第一个本地管理员在创建时会带有一个标记，指示他们**尚未**下载恢复密钥。

在该标记清除之前，Zerobyte 会将用户重定向到 `/download-recovery-key`，而不是正常的应用页面。

当用户成功下载文件时：

- Zerobyte 验证用户的当前密码
- 响应返回明文 `restic.pass` 文件
- Zerobyte 将该用户标记为已下载恢复密钥

之后，未来的登录将直接进入应用。

## SSO 行为

SSO 创建的用户被标记为已下载恢复密钥，因此他们**不会**在首次登录时被强制通过恢复密钥下载屏幕。

这意味着 SSO 部署应将恢复密钥存储视为组织管理员的责任。普通 SSO 用户不会单独被提示保存密钥。

## 谁可以下载恢复密钥

当前的下载端点需要满足以下所有条件：

- 您已登录
- 您在当前活跃组织中的角色为 `admin` 或 `owner`
- 您重新输入当前账户密码

下载的文件名为 `restic.pass`。

## 新建仓库与导入仓库

| 仓库场景 | Zerobyte 使用的密码 | 在 UI 中选择什么 |
| --- | --- | --- |
| 在 Zerobyte 中创建的新仓库 | 当前活跃组织的恢复密钥 | 不显示额外的密码选择 |
| 已在使用当前活跃组织恢复密钥的现有仓库 | 当前活跃组织的恢复密钥 | `导入现有仓库` + `使用现有的恢复密钥` |
| 在其他地方使用不同 Restic 密码创建的现有仓库 | 每个仓库的自定义密码 | `导入现有仓库` + `手动输入密码` |

### 重要限制

自定义密码选项仅在启用**导入现有仓库**时存在。Zerobyte 创建的全新仓库始终使用当前活跃组织的恢复密钥。

## Zerobyte 如何决定使用哪个密码

Zerobyte 的运行时密码选择很简单：

1. 如果仓库是导入的**并且**存储了自定义密码，则使用该自定义密码。
2. 否则，使用当前活跃组织的恢复密钥。

这有一个重要后果：下载组织恢复密钥**无法**帮助访问配置了自定义密码的导入仓库。

## 选择正确的导入选项

仅当仓库已经使用与当前活跃组织相同的 Restic 密码时，选择**使用现有的恢复密钥**。

典型示例：

- 之前由同一 Zerobyte 组织创建的仓库
- 迁移环境中组织密码被保留的情况

当仓库是在此组织外部初始化时选择**手动输入密码**，例如：

- 使用 Restic CLI 创建的仓库
- 由另一个 Zerobyte 组织创建的仓库
- 由其他恰好使用 Restic 的工具创建的仓库

## 实用指导

- 保留使用自定义密码的导入仓库的安全清单。
- 不要假设同一 Zerobyte 实例中的所有仓库共享相同的密码模式。
- 如果您管理多个组织，请记住下载的恢复密钥是针对**当前活跃**组织的。
- 使用仓库实际使用的相同密码模式测试恢复。
- 将 `APP_SECRET` 和恢复密钥视为不同的运营密钥。丢失一个不等于丢失另一个。
