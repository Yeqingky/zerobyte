Zerobyte 的通知系统分为两层：

1. **通知目的地** 是一个可复用的通知渠道，例如 Slack、SMTP、ntfy 或 Webhook。
2. **备份计划分配** 决定哪些事件应发送到哪个目的地。

您可以在 **通知** 下创建目的地，并在备份详情页面上为每个备份计划配置路由。

## Zerobyte 的通知工作方式

- 目的地按组织范围进行管理。
- 敏感的目的地字段在存储前会加密。
- 已禁用的目的地会保留，但 Zerobyte 在发送通知时会跳过它们。
- **测试** 按钮会向该目的地发送一条通用测试消息。
- 每个计划的路由支持四个事件：`start`（开始）、`success`（成功）、`warning`（警告）和 `failure`（失败）。

<Callout type="info">
  目的地只需创建一次，之后可以在任意数量的备份计划中重复使用。
</Callout>

## 支持的目的地类型

| 类型 | 适用场景 | 关键字段 |
| --- | --- | --- |
| `Email (SMTP)` | 现有邮件基础设施、多个收件人、工单别名 | SMTP 主机、端口、发件人、收件人、可选认证、TLS |
| `Slack` | 团队聊天频道 | Webhook URL、可选机器人用户名、可选图标表情 |
| `Discord` | 服务器频道或子线程 | Webhook URL、可选用户名、可选头像 URL、可选线程 ID |
| `Gotify` | 自托管的移动端或桌面端推送 | 服务器 URL、应用令牌、优先级、可选路径 |
| `ntfy` | 轻量级的发布-订阅通知 | 主题、可选服务器 URL、可选认证、优先级 |
| `Pushover` | 个人手机提醒 | 用户密钥、API 令牌、可选设备、优先级 |
| `Telegram` | 群组或私聊通知 | 机器人令牌、聊天 ID、可选线程 ID |
| `Generic Webhook` | 任意 HTTP 端点和自动化系统 | URL、方法、请求头、内容类型、可选 JSON 模板键 |
| `Custom (Shoutrrr URL)` | 高级用户或内置表单未覆盖的服务 | 原始 Shoutrrr URL |

<Callout type="warn">
  出站 HTTP 通知目的地必须使用在 `WEBHOOK_ALLOWED_ORIGINS` 中列出的来源。这适用于 Generic Webhook、Gotify、自托管 ntfy 服务器以及解析到自定义 HTTP 端点的 Custom Shoutrrr URL。
</Callout>

## 应该使用哪种类型？

- 当您已经信任某个 SMTP 中继并希望轻松向多个收件人发送通知时选择 **Email**。
- 当备份事件需要发送到共享团队空间时选择 **Slack** 或 **Discord**。
- 当您更喜欢自托管的推送服务时选择 **Gotify** 或 **ntfy**。
- 当您希望在手机上快速接收个人提醒时选择 **Pushover** 或 **Telegram**。
- 当另一个系统期望收到普通 HTTP 请求时选择 **Generic Webhook**。
- 当内置类型接近但不够灵活，或您需要 Shoutrrr 支持的其他服务时选择 **Custom (Shoutrrr URL)**。

## 创建目的地

1. 打开侧边栏中的 **通知**。
2. 点击 **创建目的地**。
3. 选择通知类型。
4. 填写该类型对应的设置项。
5. 保存目的地。
6. 打开目的地并使用 **测试** 功能，然后再将其挂接到生产环境中。

在当前 UI 中，类型在创建后实际上是固定的，因此请创建新的目的地，而不是试图将现有的 Slack 修改为 Email 等。

## 按备份计划路由通知

1. 打开要配置的备份计划。
2. 在 **通知** 部分，点击 **添加通知**。
3. 选择一个现有的目的地。
4. 选择哪些事件应触发该目的地。
5. 保存备份计划的通知设置。

每个分配是独立的。一个计划可以将失败事件发送到 Email，将警告发送到 Slack，而成功事件不发送任何通知。

## 备份事件含义

| 事件 | Zerobyte 何时发送 | 包含内容 |
| --- | --- | --- |
| `start` | 备份开始运行时 | 卷名称、仓库名称和计划名称 |
| `success` | 备份成功完成 | 卷、仓库、计划，以及可用时的备份摘要详情 |
| `warning` | 备份完成但带有警告 | 成功详情加上可用时的警告文本 |
| `failure` | 备份验证或执行失败 | 卷、仓库、计划和错误文本 |

  <Tab value="Email (SMTP)">
    当您希望将通知发送给相关人员、共享邮箱或已处理邮件的下游系统时，使用 Email。

    **字段**

    - `SMTP Host`（SMTP 主机）
    - `SMTP Port`（SMTP 端口）
    - `Username` 和 `Password`（可选）
    - `From Address`（发件人地址）
    - `From Name`（发件人名称，可选）
    - `To Addresses`（收件人地址）
    - `Use TLS`（是否使用 TLS）

    **说明**

    - `To Addresses` 在 UI 中是以逗号分隔的列表。
    - TLS 是简单的开关选项。
    - SMTP 认证是可选的，因为某些中继接受受信任的网络发送方。

    **示例**

    ```text
    Type: Email (SMTP)
    SMTP Host: smtp.example.com
    SMTP Port: 587
    Username: alerts@example.com
    Password: <smtp-password>
    From Address: alerts@example.com
    From Name: Zerobyte Alerts
    To Addresses: ops@example.com, backups@example.com
    Use TLS: On
    ```
  </Tab>

  <Tab value="Slack">
    在团队频道中使用 Slack 以实现共享的操作可见性。

    **字段**

    - `Webhook URL`
    - `Bot Username`（可选）
    - `Icon Emoji`（可选）

    **示例**

    ```text
    Type: Slack
    Webhook URL: https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXX
    Bot Username: Zerobyte
    Icon Emoji: :floppy_disk:
    ```
  </Tab>

  <Tab value="Discord">
    当您的团队在服务器频道中协作，或希望向特定线程发布消息时，使用 Discord。

    **字段**

    - `Webhook URL`
    - `Bot Username`（可选）
    - `Avatar URL`（可选）
    - `Thread ID`（可选）
  </Tab>

  <Tab value="Gotify">
    当您希望使用自托管推送并明确控制优先级时，使用 Gotify。

    **字段**

    - `Server URL`（服务器 URL）
    - `App Token`（应用令牌）
    - `Priority` 从 `0` 到 `10`
    - `Path`（可选）

    **说明**

    - 将 Gotify 服务器来源（如 `https://gotify.example.com`）添加到 `WEBHOOK_ALLOWED_ORIGINS`。
  </Tab>

  <Tab value="ntfy">
    当您需要一个简单的基于主题的推送服务（无论是在 `ntfy.sh` 还是您自己的服务器上）时，使用 ntfy。

    **字段**

    - `Server URL`（可选，留空则使用 `ntfy.sh`）
    - `Topic`（主题）
    - `Username` 和 `Password`（可选）
    - `Access token`（访问令牌，可选）
    - `Priority`（优先级）

    **说明**

    - 将 `Server URL` 留空以使用 `ntfy.sh`。
    - 将自托管 ntfy 的来源（如 `https://ntfy.example.com`）添加到 `WEBHOOK_ALLOWED_ORIGINS`。
    - 如果设置了访问令牌，它将优先于用户名和密码。
    - 优先级值为 `max`、`high`、`default`、`low` 和 `min`。

    **示例**

    ```text
    Type: Ntfy
    Server URL: https://ntfy.example.com
    Topic: zerobyte-backups
    Access token: <token>
    Priority: high
    ```
  </Tab>

  <Tab value="Pushover">
    使用 Pushover 接收直接的个人推送通知。

    **字段**

    - `User Key`（用户密钥）
    - `API Token`（API 令牌）
    - `Devices`（可选）
    - `Priority` 值为 `-1`、`0` 或 `1`
  </Tab>

  <Tab value="Telegram">
    当您希望在机器人聊天或群组主题中接收通知时，使用 Telegram。

    **字段**

    - `Bot Token`（机器人令牌）
    - `Chat ID`（聊天 ID）
    - `Thread ID`（可选）

    **示例**

    ```text
    Type: Telegram
    Bot Token: 123456789:ABC-DEF1234ghIkl-zyx57W2v1u123ew11
    Chat ID: -1231234567890
    Thread ID: 3
    ```
  </Tab>

  <Tab value="Generic Webhook">
    当另一个服务期望接收普通 HTTP 请求而非特定提供商的 webhook 格式时，使用 Generic Webhook。

    **字段**

    - `Webhook URL`
    - `Method` 为 `GET` 或 `POST`
    - `Content Type`（内容类型）
    - `Headers`（请求头）
    - `Use JSON Template`（是否使用 JSON 模板）
    - `Title Key` 和 `Message Key`（启用 JSON 模式时）

    **说明**

    - 将 webhook 来源（如 `https://hooks.example.com`）添加到 `WEBHOOK_ALLOWED_ORIGINS`。
    - 请求头按每行一个 `Key: Value` 的格式输入。
    - 启用 JSON 模式后，Zerobyte 会发送一个类似 `{titleKey: "...", messageKey: "..."}` 格式的请求体。
    - 禁用 JSON 模式时，请求体为纯通知文本。

    **示例**

    ```text
    Type: Generic Webhook
    Webhook URL: https://hooks.example.com/backup-events
    Method: POST
    Content Type: application/json
    Headers:
    Authorization: Bearer <token>
    X-Source: zerobyte
    Use JSON Template: On
    Title Key: title
    Message Key: message
    ```
  </Tab>

  <Tab value="Custom">
    当您已经知道要让 Zerobyte 使用的确切 Shoutrrr URL 时，使用 Custom。

    请参阅 [Shoutrrr 文档](https://shoutrrr.nickfedor.com/services/overview) 了解支持的 URL 格式和服务特定选项。

    如果 Shoutrrr URL 指向自定义 HTTP 端点，请将该端点来源添加到 `WEBHOOK_ALLOWED_ORIGINS`。

    **字段**

    - `Shoutrrr URL`

    **示例**

    ```text
    slack://hook:T000-B000-XXX@webhook?username=Zerobyte
    ```
  </Tab>
</Tabs>

## 实用的路由模式

| 目标 | 建议路由 |
| --- | --- |
| 仅在出现真正问题时提醒相关人员 | `failure` 发送到 Email 或 Pushover |
| 让团队了解情况而不产生噪音 | `warning` 和 `failure` 发送到 Slack 或 Discord |
| 审计每次运行 | `start`、`success`、`warning` 和 `failure` 发送到 Email 或 Generic Webhook |
| 以移动端为主的个人提醒 | `failure` 发送到 Telegram、ntfy、Gotify 或 Pushover |
| 接入其他事件或工作流系统 | `failure` 以及可选的 `warning` 发送到 Generic Webhook |
